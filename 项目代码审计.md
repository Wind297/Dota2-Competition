### 1.2 算分核心数据流（`matches.py`）

```
create_match             → 仅落 Match + 10 个 MatchPlayer(is_winner=None)
submit_or_update_result  → _apply_result → _set_match_player_delta
                           （幂等：diff = target - mp.score_delta → 写 SeasonPlayer
                            current_score + ScoreAuditLog）
patch_match              → 名单变更：_clear_match_player_delta(移除者) + 新增 MatchPlayer
delete_match             → _clear_match_player_delta(全员) → 删 Match
```

```python
    skip_score=False, boss_player_id=None, is_banming=False,
) -> None:
    if is_banming:
        # 板命局：所有人 ±2，跳过老板规则和扣分名单
        ...
        return
    # 老板规则 / 默认规则 ...
```

## 二、关键约束与保护机制（已实现的"硬规则"）

| 约束 | 实现位置 | 触发条件 |
|---|---|---|
| 归档赛季不可编辑比赛 | `_ensure_active_season_match` | `season.status == archived` 时 PATCH/DELETE 比赛抛 400 |
| 已完赛不可切换 `is_banming` | `patch_match` | `body.is_banming != m.is_banming and m.status == completed` 抛 400 |
| 一场比赛最多 1 名"老板" | `submit_or_update_result` | 名字含"老板"的选手 > 1 时抛 400 |
| 扣分阈值校验（防前端绕过） | `submit_or_update_result` | `deduct_set` 中任一选手基础分 ≤ threshold 抛 400 |
| 必须恰好 5 名胜者 | `submit_or_update_result` | `len(set(winners)) != 5` 抛 400 |
| 胜者与扣分名单不可重叠 | `submit_or_update_result` | `win_set & deduct_set` 抛 400 |
| 比赛日冲突检查 | `create_match` | 选手在本比赛日已有未完赛比赛抛 400 |
| 名单变更时新加入选手冲突检查 | `patch_match` | 同上 |
| 创建/编辑比赛选手必须在当前赛季 active | `create_match`/`patch_match` | `SeasonPlayer.is_active=True` 校验 |
| 当前赛季无活跃时拒绝写比赛/选手 | `get_active_season` 抛 400 | 各 router 调用 |
| 积分审计幂等 | `_set_match_player_delta` 用 diff 写差量 | 重复提交不重复加分 |
| `recalculate-scores` 不写审计 | `seasons.py` 第 235 行注释 | 重置是元操作，差额由前端 `scoreDiff` 提示 |
| 板命局跳过老板识别和扣分阈值校验 | `submit_or_update_result` | `m.is_banming=true` 时 `boss_candidates=[]`、跳过阈值循环 |

## 三、客观限制与潜在风险（按严重程度排序）

> 状态列含义：🔴 未修 / 🟡 已知 / 🟢 已修 / ⚪ 待确认

### 🔴 严重（建议尽快处理）

#### 1. `guest-token` 等同于 admin token，后端无角色隔离 🔴

`backend/app/routers/auth.py:25-29`

```python
@router.get("/guest-token", response_model=LoginResponse)
def guest_token():
    """游客只读 token，无需密码，前端负责限制写操作。"""
    token = create_access_token()  # ← 与 admin login 走的是同一个 serializer.dumps
    return LoginResponse(access_token=token)
```

`require_auth` 只校验 token 签名有效性，不区分 role。

**实际效果**：任何人打开网页 → 前端调 `/api/auth/guest-token` 拿到 token → 这把 token 可以直接调所有写接口（创建比赛、删除比赛、改积分等）。前端所谓"游客只读"完全靠 `v-if="adminMode"` 显隐，抓包就能绕过。

#### 2. `secret_key` 默认值是公开字符串 🔴

`backend/app/config.py:7-13`

```python
class Settings(BaseSettings):
    secret_key: str = "change-me-in-production"
    admin_password: str = "666"
```

如果生产 `.env` 没设置 `secret_key`，任何人都能用这个公开值自己签合法 admin token，绕过密码登录。

#### 3. `is_practice` 切换**没有完赛保护**（与 `is_banming` 形成对比） 🔴

`backend/app/routers/matches.py:244-245`

```python
if body.is_practice is not None:
    m.is_practice = body.is_practice
```

可以已完赛后把练习赛改成正式赛（或反过来），但 `score_delta` 不会自动重算，下次名单变更时 `_clear + _apply_result` 会按新 `is_practice` 走 `skip_score` 分支，悄悄加减分。这是和当时讨论 `is_banming` 完赛保护时同一个问题，但 `is_practice` 没补上对应保护。

#### 4. `daily_reset` 在赛季体系下失效 🔴

`backend/app/daily_reset.py:12-38`

`maybe_reset_daily_online` 更新的是 `Player.is_online`，但新代码所有读路径用的是 `SeasonPlayer.is_online`（`players.py` `_player_to_out`）。`Player.is_online` 是 `models.py:49-52` 标注的"历史字段，新代码不再使用"。

**实际效果**：每个比赛日的"全员重新上线"功能**实际已经不工作了**——前端看到的"在线状态"是 `SeasonPlayer.is_online`，从不被批量重置，只能靠管理员手动 `patch_player` 一名一名改。

### 🟡 中等（影响数据一致性或体验）

#### 5. `PlayerDetailModal` 不接收 `seasonId`，跨赛季查看排行榜时数据错位 🟡

`frontend/src/components/PlayerDetailModal.vue:163`

```ts
scoreHistory.value = await fetchPlayerScoreHistory(pid);
```

`RankingsView` 允许切换历史赛季查看排行榜，但点开选手后 Modal 不传 `seasonId`，永远拉当前赛季的积分明细。用户在"第 1 赛季"排行榜点选手，看到的是"当前赛季"的明细。

#### 6. `recalculate_season_scores` 不写审计差额，前端 `scoreDiff` 长期挂红 🟡

`backend/app/routers/seasons.py:230-263`

`PlayerDetailModal` 的 `scoreDiff = current_score - historySum` 在重置后会显示差额（因为旧 `manual_adjust` 审计记录仍在但当前分已被覆盖），且没有"重置"审计记录可追溯。

#### 7. `_has_column` / `_safe_alter` 在非 SQLite 上不可靠 🟡

`backend/app/main.py:34-51`

PRAGMA 是 SQLite 专属，PostgreSQL/MySQL 上会抛异常被 except 吞掉返回 True，然后 `_safe_alter` 又把 ALTER 失败吞掉，迁移在非 SQLite 上"看起来成功"实际可能没生效。当前部署用 SQLite 没问题，但若迁 PostgreSQL 会踩坑。

#### 8. `list_players` / `fetchMatches` / `get_player_score_history` 都没分页 🟡

- `players.py` `select(SeasonPlayer, Player)` 一次拉全部，再对每个选手做 4 次聚合 lookup
- `matches.py` `select(Match)` 一次拉全部，且每行调 `_compute_sequence_no` 做一次 COUNT 查询（N+1）
- `players.py` `select(ScoreAuditLog)` 一次拉选手全部历史记录

选手 ~50 人、比赛 ~100 场时无感；超过 200/500 量级会明显变慢。

#### 9. `set_final_ranks` 有死代码 🟡

`backend/app/routers/seasons.py:177-186`

```python
# 先清除该赛季所有已有名次
db.execute(
    select(SeasonPlayer)
    .where(SeasonPlayer.season_id == season_id, SeasonPlayer.final_rank.isnot(None))
)  # ← 这个 select 没赋值也没 delete，是 dead code
from sqlalchemy import update
db.execute(
    update(SeasonPlayer)
    .where(SeasonPlayer.season_id == season_id)
    .values(final_rank=None)
)
```

第一条 `select` 完全没用，应该删掉。

#### 10. `_ensure_active_season_match` 用 `__import__` 内联导入 🟡

`backend/app/routers/matches.py:61-67`

```python
def _ensure_active_season_match(m: Match, db: Session) -> None:
    if m.season_id is None:
        return
    season = db.get(__import__("app.models", fromlist=["Season"]).Season, m.season_id)
```

文件顶部已经从 `app.models` 导入了 `Match, MatchPlayer, ...`，唯独 `Season` 没导入。应直接 `from app.models import Season`。

#### 11. CORS 配置自相矛盾 🟡

`backend/app/main.py:16-22`

```python
app.add_middleware(
    CORSMiddleware,
    allow_origins=["*"],
    allow_credentials=True,  # ← 浏览器规范禁止 * + credentials=True
    allow_methods=["*"],
    allow_headers=["*"],
)
```

按 CORS 规范，`allow_credentials=True` 时 `allow_origins` 不能是 `*`。所幸前后端同源部署，CORS 不生效，但配置是错的。

### 🟢 轻度（不影响功能但属于代码异味）

#### 12. `MatchPlayer.is_winner` 是 `bool | None` 但代码每次都设置 🟢

`models.py:125` nullable，语义上想表达"未提交结果"，但 `_apply_result` 第 359、378、385 行每次都强制赋值。`Match.status` 已经能区分"已确认/已完赛"，`is_winner=None` 这个状态实际用不到。

#### 13. 老板识别靠字符串 `"老板" in name` 🟢

`matches.py:443`。如果有选手叫"小老板"、"老板娘"也会被识别为老板触发 ±2 老板规则。当前已有"一场最多 1 名老板"的二次校验防多老板冲突，但语义上不一定符合预期。

#### 14. `_compute_sequence_no` 不校验手工 `sequence_no` 冲突 🟢

`matches.py:70-83`：手工指定后直接返回，不查重。可能两场比赛都显示"第 3 场"。

#### 15. `bulk_import_players` 中 `db.flush()` 在 try 外 🟢

`players.py:271-288`。新建选手时 `db.flush()` 抛 `IntegrityError` 不会被内层捕获，会冒到外层 try/except 整体 rollback。功能上 OK，但风格不一致。

#### 16. `ScoreAuditLog` 缺 `created_by` 字段 🟢

无法追溯是哪个管理员做了 `manual_adjust`。如果有多名管理员，无法追责。同样 `Match` 也没 `created_by`。

#### 17. 历史遗留字段双重存储 🟢

`Player.is_online` / `Player.current_score` 在 `models.py:49-52` 标注"历史字段"，但 `daily_reset` 还在更新 `Player.is_online`，`create_player` 创建 `Player()` 时这两个字段用默认值 0/True。新人接手会困惑"为什么 is_online 改了前端没反映"。

#### 18. 前端 `MatchDetailView` 仍有未使用的 `NTag` import 🟢

`vue-tsc` 报的 `MatchDetailView.vue(13,3): error TS6133: 'NTag' is declared but its value is never read`，是先前遗留问题。
