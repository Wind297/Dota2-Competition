Dota2 个人积分赛管理系统 v2.0 — 在线网站方案
1. 产品目标
提供一个无需本地安装、通过浏览器即可访问的管理后台，让比赛管理人员可以：

一键查看所有选手及当日状态

用筛选器/快捷方案精准定位可参赛选手

创建比赛、确认上场名单、记录胜负结果

查看积分排名和历史比赛
全程无需编写代码或配置环境。

2. 用户（管理员）使用流程
2.1 首页仪表盘
打开网站后进入选手池页面，默认显示全部选手，包含：

选手名、当前积分、今日参赛场数、今日胜场数、在线/离线标记

顶部区域：快捷方案按钮（高分场、中分场等）

左侧或顶部可展开高级筛选器（积分区间、今日状态、在线状态）

2.2 赛前筛选
管理员点击快捷按钮（如“中分场”），系统自动应用对应筛选条件，只展示符合资格的选手。

若需微调，管理员可手动调整高级筛选器中的选项，列表实时更新。

列表中的离线选手灰色显示，管理员可通过点击头像/按钮标记某选手“在线”或“离线”。

2.3 创建比赛与确认名单
管理员在筛选后的列表中勾选10位选手。

点击“创建比赛”按钮，系统校验10人是否满足并发规则，成功后自动新建一场比赛，状态为“已确认”。

进入比赛详情页，列出10名选手，等待他们完成游戏。

2.4 录入结果
比赛结束后，管理员在该场比赛详情页勾选5名胜者，点击“提交结果”。

系统提示确认，提交后积分自动更新，页面跳转回选手池。

2.5 查看历史与排行
顶部菜单可切换到“排行榜”页面，查看按积分降序排列的选手。

“历史比赛”页面可按日期或状态筛选，查看以往每场的参与者和胜负。

3. 前端页面设计
3.1 页面结构
导航栏：顶部固定，包含“选手池”、“历史比赛”、“排行榜”三个标签。

选手池页面：

左侧/顶部筛选区：快捷方案按钮组（可单选）、高级筛选面板（可折叠）

选手卡片/表格：显示姓名、积分、今日数据、在线状态开关

底部固定操作栏：显示已选人数（需恰好10人），以及“创建比赛”按钮

比赛详情页：从历史比赛列表中点击进入，展示参与者头像/名称，赛后勾选胜者提交结果。

排行榜页面：纯表格，积分从高到低。

3.2 交互细节
在线/离线切换：选手项右侧有一个开关，点击即可切换，无需保存按钮。

筛选实时生效：高级筛选的每个条件变化，列表立即刷新（防抖请求）。

已选人数计数器：始终可见，不足或超过10人时按钮禁用并给出提示。

响应式设计：支持手机和电脑浏览器，方便管理员在比赛现场用手机操作。

4. 后端架构（不改动核心逻辑，只做适配）
沿用之前已设计好的 API，但需要增加少量对前端友好的接口，如批量获取快捷方案对应的筛选参数等。技术栈不变：

Python FastAPI

SQLAlchemy + SQLite 或 PostgreSQL

部署在云平台

5. 部署方案（零运维代价）
选择 Railway 或 Render 等免费/低成本 PaaS：

后端服务：通过 GitHub 仓库自动部署 FastAPI 应用

数据库：使用 Railway 或 Render 提供的免费层 PostgreSQL/MySQL，或云服务商的免费 MySQL（如 TiDB Cloud，Supabase）

前端：可以直接和后端放在同一个服务里提供静态文件，或托管于 Vercel / Netlify

最终交付物：

一个公开的 URL，如 https://dota2-tournament.railway.app

管理员访问该 URL 即进入管理后台，所见即所得

6. 开发计划（分步实现）
第一步：基础后端与 API
搭建 FastAPI 项目，完成数据库模型、CRUD、筛选接口（已有基础）

增加一些便捷接口：例如 GET /api/presets 返回快捷方案及其过滤参数

第二步：前端 SPA 开发
使用 Vue 3 + Vite（学习成本低，体积小）搭建前端项目

集成 UI 库（如 Naive UI 或 Element Plus）快速实现美观界面

主要页面：选手池（含筛选/快捷方案）、比赛创建与结果提交、排行榜、历史比赛

调用后端 API，处理状态同步

第三步：对接部署
将前端打包后的静态文件由 FastAPI 直接托管（单服务部署，方便免费）

配置数据库，设置环境变量

部署到 Railway，绑定自定义域名（可选）

7. 需要确认的事项
数据库选型：如果只是管理员团队内部用（<10人并发），SQLite 就足够，且部署最简单；如需多设备同时写，可用 PostgreSQL 免费层。请告诉我你的倾向。

用户认证：目前是否为完全开放访问，还是需要一个简单的管理员密码登录？（v1 可以不加，后续再加）

域名或命名：你可以自己准备一个域名，或直接用平台生成的二级域名，不影响使用。
